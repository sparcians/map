This chapter collects modeling patterns and practices that experienced
Sparta engineers apply consistently.  Some are about correctness --
patterns that prevent subtle bugs.  Others are about clarity -- making
the model easier to understand and debug.  All of them are illustrated
with concrete code.

== Modeling a Mux: The Ordering Problem

The multiplexer (mux) is the canonical example of the ordering problem
in cycle-based simulation -- and the clearest illustration of why
Sparta's event and phase system exists.

=== The Problem

Consider a mux with three data inputs (`InputA`, `InputB`, `InputC`),
a select signal, and a result output.

[plantuml,mux-diagram,png,align="center"]
....
@startuml
rectangle "InputA()" as A
rectangle "InputB()" as B
rectangle "InputC()" as C
rectangle "Select()" as S
rectangle "Result"   as R

A --> R
B --> R
C --> R
S --> R : select
@enduml
....

The constraint is:

- You cannot compute `Result` until `Select` has been called
- You cannot call `Select` until all inputs (`InputA`, `InputB`,
  `InputC`) are available

In a 0-cycle turnaround model (where all inputs arrive in the same
cycle and the result must also be produced in the same cycle), this
creates an ordering problem.  If the three input callbacks and the
select callback can fire in any order, the model may compute the
result before all inputs have arrived.

SystemC solves this with delta cycles -- borrowed from VHDL -- but
delta cycles are limited to a finite upper bound and do not scale
cleanly to complex topologies.

Sparta solves it with explicit event ordering via the DAG and
scheduling phases.  The solution is deterministic, has no iteration
limit, and is expressed directly in the model code.

=== The Solution

The key insight is that `Select` is an *event*, not a port callback.
The three input port callbacks each store their value and schedule
the same `UniqueEvent` (`evSelect`).  Because `evSelect` is a
`UniqueEvent`, it fires exactly once per cycle regardless of how many
inputs arrive.  The ordering is enforced with
`registerConsumingEvent` and `registerProducingEvent`, which tell the
DAG that `evSelect` must fire *after* all three input ports have
delivered their data, and that `pResult` is driven *only after*
`evSelect` fires.

==== Components

[source,cpp]
----
// Three 0-cycle data input ports
sparta::DataInPort<Value> pInputA_{&unit_port_set_, "pInputA", 0};
sparta::DataInPort<Value> pInputB_{&unit_port_set_, "pInputB", 0};
sparta::DataInPort<Value> pInputC_{&unit_port_set_, "pInputC", 0};

// The select event -- UniqueEvent fires only once per cycle
sparta::UniqueEvent<> evSelect_{&unit_event_set_, "evSelect",
                                 CREATE_SPARTA_HANDLER(MuxUnit, select_)};

// The result output port
sparta::DataOutPort<Value> pResult_{&unit_port_set_, "pResult"};
----

==== Enforcing the Ordering

In the unit constructor, register the consuming and producing event
relationships with the ports:

[source,cpp]
----
MuxUnit::MuxUnit(sparta::TreeNode * node,
                 const MuxParameters * params) :
    sparta::Unit(node)
{
    // Register callbacks
    pInputA_.registerConsumerHandler(
        CREATE_SPARTA_HANDLER_WITH_DATA(MuxUnit, inputA_, Value));
    pInputB_.registerConsumerHandler(
        CREATE_SPARTA_HANDLER_WITH_DATA(MuxUnit, inputB_, Value));
    pInputC_.registerConsumerHandler(
        CREATE_SPARTA_HANDLER_WITH_DATA(MuxUnit, inputC_, Value));

    // Tell the DAG: evSelect_ must fire AFTER all three input ports
    // have delivered their data.  This establishes a happens-before
    // relationship: port delivery -> evSelect_.
    pInputA_.registerConsumingEvent(evSelect_);
    pInputB_.registerConsumingEvent(evSelect_);
    pInputC_.registerConsumingEvent(evSelect_);

    // Tell the DAG: pResult_ is driven only AFTER evSelect_ fires.
    // This establishes: evSelect_ -> pResult_ send.
    pResult_.registerProducingEvent(evSelect_);
}
----

`registerConsumingEvent(ev)` tells the DAG that the event `ev` must
fire after this port's callback fires within the same cycle.

`registerProducingEvent(ev)` tells the DAG that data sent on this
port is driven by event `ev` -- any consumer of `pResult_` will
be guaranteed to receive data only after `evSelect_` has run.

==== The Callbacks

[source,cpp]
----
// The input callbacks -- store the value and schedule the selector
void MuxUnit::inputA_(const Value & val) {
    a_value_ = val;
    evSelect_.schedule();  // schedule evSelect -- fires only once
}

void MuxUnit::inputB_(const Value & val) {
    b_value_ = val;
    evSelect_.schedule();  // already scheduled? UniqueEvent ignores it
}

void MuxUnit::inputC_(const Value & val) {
    c_value_ = val;
    evSelect_.schedule();  // same
}

// The selector -- called once per cycle, after all inputs have arrived
void MuxUnit::select_() {
    result_ = chooseWinner_();
    pResult_.send(result_);  // can be 0 or more cycles -- legal!
}
----

The three input callbacks each schedule `evSelect_`.  Because
`evSelect_` is a `UniqueEvent`, it fires exactly once regardless of
whether one, two, or three inputs arrived this cycle.  When
`select_()` runs, all three values are guaranteed to have been
stored because the DAG enforces that all three port callbacks
complete before `evSelect_` fires.

==== Why This Works

The ordering guarantee comes from two cooperating mechanisms:

1. **`registerConsumingEvent`** adds a DAG edge: port callback →
   `evSelect_`.  The scheduler will not fire `evSelect_` until all
   three port forward events (one per input port) have completed.

2. **`UniqueEvent`** ensures that `select_()` fires exactly once even
   though three different callbacks each call `evSelect_.schedule()`.

Together they give a clean, deterministic solution: all inputs settle
first, then the select logic runs exactly once, then the result is
sent.  There is no iteration, no delta cycle limit, and no risk of
the result being computed before all inputs have arrived.

=== Contrast with the Broken Pattern

To understand why the ordering registration matters, consider what
happens without it:

[source,cpp]
----
// BROKEN -- do not do this
void MuxUnit::inputA_(const Value & val) {
    a_value_ = val;
    result_ = chooseWinner_();  // InputB and InputC may not have arrived yet!
    pResult_.send(result_);     // Result may be wrong
}
----

This processes the result in the port callback directly.  If
`InputA`'s callback fires before `InputB` and `InputC`, the result
is computed with stale or uninitialized values for B and C.  The
bug is non-deterministic -- it may produce correct results in some
runs and wrong results in others depending on the order the scheduler
happens to fire the port callbacks.

Even this slightly better version is broken:

[source,cpp]
----
// STILL BROKEN -- do not do this
void MuxUnit::inputA_(const Value & val) {
    a_value_ = val;
    evSelect_.schedule();  // schedule the event...
}
// ... same for B and C ...

void MuxUnit::select_() {
    result_ = chooseWinner_();
    pResult_.send(result_);
}
// But WITHOUT registerConsumingEvent -- no DAG edge established
----

Without the `registerConsumingEvent` calls, the DAG has no edge
between the port callbacks and `evSelect_`.  The scheduler may fire
`evSelect_` before all three input callbacks have run, again
producing wrong results.

The `registerConsumingEvent` + `registerProducingEvent` calls are not
optional.  They are what tells Sparta about the data dependencies your
model has, so the framework can enforce them.

=== Complete MuxUnit Example

Here is the full, correct implementation:

[source,cpp]
----
// MuxUnit.hpp
#pragma once
#include "sparta/simulation/Unit.hpp"
#include "sparta/ports/DataPort.hpp"
#include "sparta/events/UniqueEvent.hpp"
#include "sparta/utils/ValidValue.hpp"

// Value type for the mux -- replace with your actual data type
using Value = uint32_t;

class MuxUnit : public sparta::Unit
{
public:
    class MuxParameters : public sparta::ParameterSet
    {
    public:
        MuxParameters(sparta::TreeNode * n) : sparta::ParameterSet(n) {}
        PARAMETER(std::string, select_policy, "priority",
                  "Mux select policy: priority, round_robin")
    };

    static const char * name;
    MuxUnit(sparta::TreeNode * node, const MuxParameters * params);

private:
    // Input ports (0-cycle -- all inputs arrive in the same cycle)
    sparta::DataInPort<Value> pInputA_{&unit_port_set_, "pInputA", 0};
    sparta::DataInPort<Value> pInputB_{&unit_port_set_, "pInputB", 0};
    sparta::DataInPort<Value> pInputC_{&unit_port_set_, "pInputC", 0};

    // The select event
    sparta::UniqueEvent<> evSelect_{&unit_event_set_, "evSelect",
                                    CREATE_SPARTA_HANDLER(MuxUnit, select_)};

    // Output port
    sparta::DataOutPort<Value> pResult_{&unit_port_set_, "pResult"};

    // Input storage
    sparta::utils::ValidValue<Value> a_value_;
    sparta::utils::ValidValue<Value> b_value_;
    sparta::utils::ValidValue<Value> c_value_;

    Value result_ = 0;
    const std::string select_policy_;

    // Callbacks
    void inputA_(const Value & val);
    void inputB_(const Value & val);
    void inputC_(const Value & val);
    void select_();

    Value chooseWinner_() const;
};
----

[source,cpp]
----
// MuxUnit.cpp
#include "MuxUnit.hpp"

const char * MuxUnit::name = "mux";

MuxUnit::MuxUnit(sparta::TreeNode * node,
                 const MuxParameters * params) :
    sparta::Unit(node),
    select_policy_(params->select_policy)
{
    // Register port callbacks
    pInputA_.registerConsumerHandler(
        CREATE_SPARTA_HANDLER_WITH_DATA(MuxUnit, inputA_, Value));
    pInputB_.registerConsumerHandler(
        CREATE_SPARTA_HANDLER_WITH_DATA(MuxUnit, inputB_, Value));
    pInputC_.registerConsumerHandler(
        CREATE_SPARTA_HANDLER_WITH_DATA(MuxUnit, inputC_, Value));

    // Establish DAG ordering:
    // All three input port callbacks must complete before evSelect_ fires
    pInputA_.registerConsumingEvent(evSelect_);
    pInputB_.registerConsumingEvent(evSelect_);
    pInputC_.registerConsumingEvent(evSelect_);

    // pResult_ is driven only after evSelect_ fires
    pResult_.registerProducingEvent(evSelect_);
}

void MuxUnit::inputA_(const Value & val) {
    a_value_ = val;
    evSelect_.schedule();
}

void MuxUnit::inputB_(const Value & val) {
    b_value_ = val;
    evSelect_.schedule();
}

void MuxUnit::inputC_(const Value & val) {
    c_value_ = val;
    evSelect_.schedule();
}

void MuxUnit::select_() {
    // At this point ALL inputs that arrived this cycle have been stored.
    // chooseWinner_ selects based on the policy and which inputs are valid.
    result_ = chooseWinner_();
    ILOG("Mux select: result=" << result_
         << " a_valid=" << a_value_.isValid()
         << " b_valid=" << b_value_.isValid()
         << " c_valid=" << c_value_.isValid());

    pResult_.send(result_);

    // Clear valid flags for next cycle
    a_value_.clearValid();
    b_value_.clearValid();
    c_value_.clearValid();
}

Value MuxUnit::chooseWinner_() const {
    // Example: priority -- A > B > C
    if (select_policy_ == "priority") {
        if (a_value_.isValid()) { return a_value_.getValue(); }
        if (b_value_.isValid()) { return b_value_.getValue(); }
        sparta_assert(c_value_.isValid(), "select_() called with no valid inputs");
        return c_value_.getValue();
    }
    // ... other policies
    sparta_assert(false, "Unknown select policy: " << select_policy_);
    return 0;
}
----

=== Key Takeaways

The mux example illustrates three general principles that apply
throughout Sparta modeling:

**1. Use `UniqueEvent` for any work triggered by multiple sources.**
Whenever more than one port callback or event might trigger the same
downstream work in the same cycle, the work method should be a
`UniqueEvent`.  This guarantees the work runs exactly once regardless
of how many triggers arrived.

**2. Use `registerConsumingEvent` to declare data dependencies.**
If an event's work depends on data delivered by a port, register that
dependency explicitly.  This is not optional -- it is what gives the
DAG the information it needs to enforce correct ordering.

**3. Use `registerProducingEvent` to declare output dependencies.**
If a port sends data that was computed by an event, register that
relationship.  Downstream consumers are then guaranteed to receive
the data only after the producing event has run.

**4. Use `ValidValue` to track which inputs have arrived.**
`sparta::utils::ValidValue<T>` is safer than a raw value with a
separate `bool valid` flag.  It asserts on read if the value has
not been set, catching missing-input bugs immediately rather than
propagating garbage values silently.

These four patterns appear together throughout Olympia and any
well-written Sparta model.  When you find yourself writing a unit
that receives data from multiple sources and produces output that
depends on all of them, reach for this pattern first.
