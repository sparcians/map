This chapter walks through installing Sparta, building the skeleton
pipeline example that ships with the framework, and running it.  By
the end of the chapter you will have a working Sparta environment and
a concrete understanding of what a Sparta simulator looks like from
the outside before diving into the code.

== Prerequisites

Sparta requires a C++17-capable compiler and CMake 3.14 or later.
The following tools and libraries must be installed before building.

=== Ubuntu 22.04 / 24.04

[source,bash]
----
sudo apt-get install -y \
    cmake \
    g++ \
    libboost-all-dev \
    libyaml-cpp-dev \
    rapidjson-dev \
    libsqlite3-dev \
    libhdf5-dev \
    zlib1g-dev \
    python3-dev \
    doxygen \
    graphviz
----

==== Windows (WSL)

Install Ubuntu 22.04 or 24.04 from the Microsoft Store, open a WSL
terminal, and follow the Ubuntu instructions above.  All subsequent
commands in this chapter are identical under WSL.

=== macOS

Install https://brew.sh[Homebrew] if not already present, then:

[source,bash]
----
brew install cmake boost yaml-cpp rapidjson sqlite hdf5 python3
----

NOTE: Sparta is tested against Apple Clang 14 and later.  If your
Xcode command line tools are out of date, run
`xcode-select --install` before proceeding.

== Cloning the Repository

Sparta lives inside the MAP (Modeling Architectural Platform)
repository alongside the Helios visualization tools.  Always clone
the stable `map_v2` branch -- the `master` branch is for development
and should not be used for modeling work.

[source,bash]
----
git clone git@github.com:sparcians/map.git \
    --recursive \
    --branch map_v2
cd map
----

The `--recursive` flag is required to pull in the git submodules that
Sparta depends on.

== Building Sparta

[source,bash]
----
cd sparta
mkdir release
cd release
cmake .. -DCMAKE_BUILD_TYPE=Release
make -j$(nproc)     # Linux / WSL
# make -j$(sysctl -n hw.logicalcpu)   # macOS
----

A successful build produces `libsparta.a` in the `release/` directory.
To install Sparta system-wide so that downstream projects (like
Olympia) can find it with `find_package(Sparta)`:

[source,bash]
----
sudo make install
----

== Building the Skeleton Pipeline

The skeleton pipeline is a minimal but complete Sparta simulator
included in the MAP repository under `sparta/example/SkeletonPipeline`.
It models a simple producer/consumer system and exercises the core
Sparta classes: `Unit`, `Port`, `Event`, `Parameter`, and `Scheduler`.

[source,bash]
----
cd map/sparta/example/SkeletonPipeline
mkdir release
cd release
cmake .. -DCMAKE_BUILD_TYPE=Release
make -j$(nproc)
----

This produces the `sparta_skeleton` binary in the `release/` directory.

== Walking Through the Code

Now that you have seen the skeleton pipeline running, it is worth
understanding what each piece of code does before the next chapters
cover each concept in depth.

The skeleton pipeline has three source files:

[cols="1,1,2"]
|===
| File | Sparta Base Class | Purpose

| `Producer.hpp` / `Producer.cpp`
| `sparta::Unit`
| Sends integers to the consumer on request

| `Consumer.hpp` / `Consumer.cpp`
| `sparta::Unit`
| Receives integers from one or more producers

| `SkeletonSimulator.hpp` / `SkeletonSimulator.cpp`
| `sparta::app::Simulation`
| Builds the tree, configures it, and binds the ports
|===

=== The Simulator: SkeletonSimulator

`SkeletonSimulator` derives from `sparta::app::Simulation` and overrides
three methods that Sparta calls in sequence during startup:

`buildTree_`:: Creates the `ResourceTreeNode` hierarchy -- one consumer
node and one or more producer nodes depending on the `num_producers`
parameter.  At the end of this phase the tree exists but no `Unit`
objects have been instantiated yet.

`configureTree_`:: An opportunity to inspect or adjust parameters before
the model is instantiated.  The skeleton pipeline does not override
this in a meaningful way, but it is available.

`bindTree_`:: Connects the ports between the producer and consumer units.
Each producer's `out_` port is bound to the consumer's corresponding
`in_` port.  This is where communication paths are established.

This three-phase construction pattern is the heart of every Sparta
simulator and is covered in full detail in
<<_simulation_construction>>.

=== The Units: Producer and Consumer

Both `Producer` and `Consumer` derive from `sparta::Unit`.  Each unit
declares its ports and events as member variables, and registers
callbacks that Sparta calls when data arrives or events fire.

The `Producer` works like this:

1. On startup, a `sparta::StartupEvent` fires the `produceData_` method
   on the first cycle.
2. `produceData_` sends a `uint32_t` value out through its
   `sparta::DataOutPort`.
3. The producer then waits for a "go" signal from the consumer via a
   `sparta::SignalInPort` before sending the next value.

The `Consumer` works like this:

1. On startup, it sends a "go" signal to each producer via a
   `sparta::SignalOutPort`, kicking off the pipeline.
2. When data arrives on its `sparta::DataInPort`, the port fires a
   callback that schedules a `sparta::UniqueEvent` for the next phase
   of the same cycle.
3. That event consumes the data and sends another "go" to keep the
   pipeline flowing.

The use of a `UniqueEvent` in the consumer -- rather than processing
data directly in the port callback -- is a deliberate pattern.  It
guarantees that the consumer processes data only after all producers
have had a chance to send in the current cycle.  This is Sparta's
event ordering system at work, and it is covered in detail in
<<_events_and_scheduling>>.

=== Ports, Parameters, and Statistics

Each unit in the skeleton pipeline also demonstrates three other core
Sparta features that appear in almost every real model:

**Parameters** are declared inside a nested `ParameterSet` class within
each unit.  They are automatically exposed on the command line and in
YAML configuration files with no extra code from the modeler.  The
`max_ints_to_send` and `num_producers` parameters you used above are
defined this way.

**Statistics** (counters) are declared as `sparta::Counter` member
variables inside the unit.  Sparta automatically collects them and
includes them in the summary output and report files you generated
earlier.

**Logging** is done through a `sparta::log::MessageSource` attached to
the unit's tree node.  Messages are emitted with `ILOG`, `WLOG`, and
`ELOG` macros and are automatically routed by the `-l` command line
option you used above.
