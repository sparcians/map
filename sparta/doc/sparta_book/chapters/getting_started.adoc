= Getting Started

This chapter walks through installing Sparta, building the skeleton
pipeline example that ships with the framework, and running it.  By
the end of the chapter you will have a working Sparta environment and
a concrete feel for what a Sparta simulator looks like from the outside
before diving into the code.

== Prerequisites

Sparta requires a C++17-capable compiler and CMake 3.14 or later.
The following tools and libraries must be installed before building.

=== Ubuntu 22.04 / 24.04

[source,bash]
----
sudo apt-get install -y \
    cmake \
    g++ \
    libboost-all-dev \
    libyaml-cpp-dev \
    rapidjson-dev \
    libsqlite3-dev \
    libhdf5-dev \
    zlib1g-dev \
    python3-dev \
    doxygen \
    graphviz
----

=== Windows (WSL)

Install Ubuntu 22.04 or 24.04 from the Microsoft Store, open a WSL
terminal, and follow the Ubuntu instructions above.  All subsequent
commands in this chapter are identical under WSL.

=== macOS

Install https://brew.sh[Homebrew] if not already present, then:

[source,bash]
----
brew install cmake boost yaml-cpp rapidjson sqlite hdf5 python3
----

NOTE: Sparta is tested against Apple Clang 14 and later.  If your
Xcode command line tools are out of date, run
`xcode-select --install` before proceeding.

== Cloning the Repository

Sparta lives inside the MAP (Modeling Architectural Platform)
repository alongside the Helios visualization tools.  Always clone
the stable `map_v2` branch -- the `master` branch is for development
and should not be used for modeling work.

[source,bash]
----
git clone git@github.com:sparcians/map.git \
    --recursive \
    --branch map_v2
cd map
----

The `--recursive` flag is required to pull in the git submodules that
Sparta depends on.

== Building Sparta

[source,bash]
----
cd sparta
mkdir release
cd release
cmake .. -DCMAKE_BUILD_TYPE=Release
make -j$(nproc)     # Linux / WSL
# make -j$(sysctl -n hw.logicalcpu)   # macOS
sudo make install
----

A successful build produces `libsparta.a` and installs it to
`/usr/local` so that downstream projects can find it with
`find_package(Sparta)`.

== Building the Skeleton Pipeline

The skeleton pipeline is a minimal but complete Sparta simulator
included in the MAP repository under `sparta/example/SkeletonPipeline`.
It models a simple producer/consumer system and exercises the core
Sparta classes: `Unit`, `Port`, `Event`, `Parameter`, and `Scheduler`.

[source,bash]
----
cd map/sparta/example/SkeletonPipeline
mkdir release
cd release
cmake .. -DCMAKE_BUILD_TYPE=Release
make -j$(nproc)
----

This produces the `sparta_skeleton` binary in the `release/` directory.

This tree output is worth studying carefully -- it is the map of
everything in the simulation.  A few things stand out:

**The tree has three phases of output.** First the Built Tree (after
`buildTree_`), then the Configured Tree (after `configureTree_`), and
finally the Finalized Tree (after elaboration).  In the Finalized Tree
you can see things that did not exist in the Built Tree: ports, events,
statistics, and log message sources.  These are created by the `Unit`
constructors during elaboration, which happens between `configureTree_`
and `bindTree_`.

**Ports show their bindings.** After `bindTree_` runs, each port shows
what it is connected to.  For example,
`top.consumer.ports.consumer_in_port` shows `[bound to]
{producer_out_port (top.producer0.ports.producer_out_port)}`.  If a
port is unbound, the tree will show it clearly -- this is one of the
fastest ways to debug a wiring mistake.

**The scheduler is a tree node too.** The global scheduler exposes its
own statistics (`ticks`, `events_fired`, etc.) and its internal event
phases (`gbl_event_Trigger`, `gbl_event_Update`, and so on) directly
in the tree.  This means you can report on scheduler statistics the
same way you report on any other statistic in the model.

**Every unit has built-in log message sources.** The `?` nodes under
each unit are log message sources for `info`, `warning`, and `debug`
categories.  The `observed:true` flag indicates which categories are
currently being captured.

=== Exploring Parameters

To see every parameter and its current value without running:

[source,bash]
----
% ./sparta_skeleton --show-parameters --no-run
----

To override a parameter on the command line -- here, telling the
producer to send 10,000 integers -- use the `-p` flag with the full
tree path to the parameter:

[source,bash]
----
% ./sparta_skeleton -p top.producer0.params.max_ints_to_send 10K \
    --show-parameters --no-run
----

The path `top.producer0.params.max_ints_to_send` maps directly to what
you saw in the tree: `top` → `producer0` → `params` →
`max_ints_to_send`.  Every parameter in every Sparta simulator is
accessible this way.

=== Using a Configuration File

For larger simulations with many parameters, managing them all on the
command line becomes unwieldy.  Sparta can write the current
configuration to a YAML file that you can edit and reuse:

[source,bash]
----
# Write the default configuration to a file
% ./sparta_skeleton --write-final-config skeleton_config.yaml --no-run

# Edit the file -- change num_producers from 1 to 100
% sed -i 's/num_producers: 1/num_producers: 100/' skeleton_config.yaml

# Run with the new configuration
% ./sparta_skeleton -c skeleton_config.yaml
----

The YAML config file approach is the standard way to set up large
simulation runs and is covered in detail in
<<_command_line_simulation>>.

=== Enabling Logging

Sparta has a built-in hierarchical logging system.  The `-l` flag takes
three arguments: a tree path pattern, a log category, and an output
file.

[source,bash]
----
# Capture all info messages from every node
% ./sparta_skeleton -l top info all.log

# Capture only consumer messages
% ./sparta_skeleton -l top.consumer info consumer.log

# Capture only producer messages (wildcard matches producer0, producer1, ...)
% ./sparta_skeleton -l top.producer* info producers.log

# Capture producer and consumer messages to the same file
% ./sparta_skeleton -l top.producer* info combined.log \
                   -l top.consumer  info combined.log
----

== Walking Through the Code

Now that you have seen the skeleton pipeline running, it is worth
understanding what each piece of code does before the next chapters
cover each concept in depth.

The skeleton pipeline has three source files:

[cols="1,1,2"]
|===
| File | Sparta Base Class | Purpose

| `Producer.hpp` / `Producer.cpp`
| `sparta::Unit`
| Sends integers to the consumer on request

| `Consumer.hpp` / `Consumer.cpp`
| `sparta::Unit`
| Receives integers from one or more producers

| `SkeletonSimulator.hpp` / `SkeletonSimulator.cpp`
| `sparta::app::Simulation`
| Builds the tree, configures it, and binds the ports
|===

=== The Simulator: SkeletonSimulator

`SkeletonSimulator` derives from `sparta::app::Simulation` and overrides
three methods that Sparta calls in sequence during startup:

`buildTree_`:: Creates the `ResourceTreeNode` hierarchy -- one consumer
node and one or more producer nodes depending on the `num_producers`
parameter.  At the end of this phase the tree exists but no `Unit`
objects have been instantiated yet.  This is what the Built Tree output
above shows.

`configureTree_`:: An opportunity to inspect or adjust parameters before
instantiation.  The skeleton pipeline uses this phase minimally, but it
is the correct place to implement topology changes that depend on
parameter values.

`bindTree_`:: Called after elaboration -- after Sparta has instantiated
all the `Unit` objects.  This is where ports are connected.
`producer_out_port` is bound to `consumer_in_port`, and
`producer0_go_port` is bound to the consumer's corresponding go port.
The bound ports you saw in the Finalized Tree output are the result of
this phase.

This three-phase construction pattern is the heart of every Sparta
simulator and is covered in full detail in
<<_simulation_construction>>.

=== The Units: Producer and Consumer

Both `Producer` and `Consumer` derive from `sparta::Unit`.  Each unit
declares its ports and events as member variables and registers
callbacks that Sparta calls when data arrives or events fire.

The `Producer` works like this:

1. A `sparta::StartupEvent` fires `produceData_` on the first cycle,
   kicking off the pipeline without waiting for an external stimulus.
2. `produceData_` sends a `uint32_t` value out through its
   `sparta::DataOutPort`.
3. The producer then waits for a "go" signal back from the consumer via
   a `sparta::SignalInPort` before sending the next value.  This
   back-pressure mechanism is what keeps the pipeline from running
   ahead of the consumer.

The `Consumer` works like this:

1. On startup, it sends a "go" signal to each producer via a
   `sparta::SignalOutPort`, priming the pipeline.
2. When data arrives on its `sparta::DataInPort`, the port fires a
   callback that schedules a `sparta::UniqueEvent` for a later phase
   of the same cycle.
3. That event consumes the data, increments `num_consumed`, and sends
   another "go" to keep the pipeline flowing.

The use of a `UniqueEvent` in the consumer -- rather than processing
data directly in the port callback -- is deliberate.  It guarantees
that the consumer processes data only after all producers have had a
chance to deliver in the current cycle, regardless of the order in
which their port callbacks fire.  This is Sparta's event ordering
system at work, and it is covered in detail in
<<_events_and_scheduling>>.

=== Parameters, Statistics, and Logging

Each unit in the skeleton pipeline also demonstrates three other core
Sparta features that appear in almost every real model:

**Parameters** are declared inside a nested `ParameterSet` class within
each unit.  They are automatically exposed on the command line and in
YAML configuration files with no extra code from the modeler.  The
`max_ints_to_send` and `num_producers` parameters you used above are
defined this way.

**Statistics** (counters) are declared as `sparta::Counter` member
variables inside the unit.  Sparta automatically collects them and
includes them in the summary output and report files you generated
earlier.  The `num_consumed` and `num_produced` counters in the tree
output are defined this way.

**Logging** is done through a `sparta::log::MessageSource` attached to
the unit's tree node.  Messages are emitted with `ILOG`, `WLOG`, and
`ELOG` macros and are automatically routed by the `-l` command line
option.  The `?` nodes in the tree output above are those message
sources.
