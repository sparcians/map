After more than two decades of building performance models and
simulation frameworks inside the semiconductor industry, three
colleagues (Knute, Dave, and Nick) sat down and listed every
requirement a modeling framework _should_ satisfy -- requirements that
existing tools like {url-systemc}[SystemC] and {url-gem5}[gem5] left
unmet.  The result, after years of design and iteration, is Sparta: a
framework built from first principles for the people who build
performance simulators for a living.

This book is the documentation that Sparta always deserved.  The book
is written by the lead designer of the framework with the goal of
turning a capable engineer into a productive Sparta modeler as quickly
as possible.  The author is the lead designer and developer of the
{url-sparcians}[Sparcians] open source organization and co-chair of
the {url-riscv-sig}[RISC-V International Performance Modeling SIG].

=== What is Sparta?
include::attributes.adoc[]

Sparta is a modeling framework and toolkit for performance architecture
engineers to build CPU, GPU, and platform models for the purpose of
performance analysis of hardware design.

Sparta is a _pure framework_, much like STL or Boost, in that Sparta
does not do anything on its own.  Instead, Sparta provides structure and
organization as well as numerous helper classes and interfaces that
allow modelers to build fast functional and performance simulators.

Sparta is built on the principles of
https://en.wikipedia.org/wiki/Software_framework[software framework design]
using https://en.wikipedia.org/wiki/Object-oriented_programming[Object-Oriented
Programming] and https://en.wikipedia.org/wiki/Design_Patterns[design
patterns].  The framework is written in {cpp} as well as Python.

Sparta was designed from the ground up to be a modeler's solution,
enabling as many dynamic, run-time features as possible.  These include
topology definitions, multiple report format generation, notification
mechanisms, event ordering, modular unit definitions, data collection
at any level at any point in time, error handling and resolution,
pipeline visualization, and more.

=== Who This Book Is For
include::attributes.adoc[]

This book is written for performance architects and modeling engineers
who want to build simulators using the Sparta framework.  Readers are
expected to have working knowledge of {cpp} and a basic
understanding of computer architecture concepts such as pipelines,
caches, and memory hierarchies.  Prior experience with another
simulation framework such as {url-systemc}[SystemC] or
{url-gem5}[gem5] is helpful but not required.

New graduate engineers will find this book most useful read from
beginning to end.  Experienced architects evaluating Sparta for a new
project may prefer to start with <<_sparta_vs_gem5>> and then
jump to the chapters most relevant to their use case.

The {url-olympia}[Olympia performance model] -- an open source
out-of-order RISC-V CPU model maintained under RISC-V International --
is a companion to this book.  Olympia is built entirely on Sparta and
serves as a complete, real-world reference for every concept covered
here.  Readers are encouraged to keep a clone of Olympia nearby and
refer to Olympia as each chapter introduces new framework concepts.

=== Sparta in the Wild
include::attributes.adoc[]

Sparta is used in both industry and academia.  Public examples include:

**{url-olympia}[Olympia]** is the canonical open source out-of-order
RISC-V CPU performance model, maintained under RISC-V International
and widely used as a starting point for RISC-V CPU performance modeling
in industry and as a teaching vehicle in graduate microarchitecture
courses.  Olympia is a trace-driven simulator modeling an RV64GC
out-of-order core, with an execution-driven mode under active
development.

**{url-coyote}[Coyote]** is an open source execution-driven simulator
from the Barcelona Supercomputing Center, built on Sparta to model
RISC-V HPC memory hierarchies.  Presented at DATE 2021, Coyote focuses
on data movement and memory system modeling, achieving simulation rates
of up to 6 MIPS across 128 simulated cores.

Beyond these public examples, Sparta has seen use at semiconductor
companies building proprietary CPU and GPU performance models where the
framework's runtime configurability, deterministic event ordering, and
rich statistics infrastructure make Sparta well suited to the demands of
pre-silicon microarchitecture exploration.


=== Sparta vs gem5 Modeling [[_sparta_vs_gem5]]
include::attributes.adoc[]

The communication and scheduling concepts in Sparta are similar in
spirit to {url-systemc}[SystemC] and {url-gem5}[gem5], but Sparta
takes a more abstract and extensible approach to modeling.

gem5 is an excellent tool for processor/system modeling and
development. gem5 is a full system simulator and can run a workload
"execution-driven style" meaning the application can run as-is on the
simulator. That's a large benefit but at the same time, a deficit to
that type of simulation platform. More on that in the following
subsections.

Sparta is a modeling framework allowing for dynamic, run-time modeling
topologies and removing the overhead of tracking time progression,
data delivery timing, data collection methodologies, explicit report
generation, etc. The models that are built using Sparta can be driven
in whatever fashion a modeler wants: agitator/instigators, traces
(like STF), or data driven like execution-driven models like gem5.

In addition, models developed in gem5 can be ported to use the Sparta
framework quite easily. Porting a model from Sparta to gem5 is more
difficult due to the lack of support for timing/event handling in
gem5, specifically 0-cycle event processing.

==== Licensing Issues

gem5 licensing is not a risk-less license, which makes some companies
leery to adopt gem5 for research/patent analysis. Sparta is based on
the Apache 2.0 license with no risks associated. Also, Sparta contains
no licensing/copyrights from ARM or companies of the like.

==== The Framework

gem5 is both a framework and a full system modeling solution. Sparta
is a framework with _examples_ of how to build custom models like the
{url-olympia}[Olympia] hosted by RISC-V International's Performance Modeling
SIG and in the examples directory.

Sparta was built from the ground up to be a modeler's solution,
enabling as many dynamic, run-time features as possible. These include
topology definitions, multiple formats report generation, notification
mechanisms. event ordering, modular unit definitions, data collection
at any level at any moment in time, error handling (and resolutions),
pipeline crawling, etc.

One advantage Sparta has over gem5 is the ability to model unit
interactions at a very detailed level especially when dealing with
0-cycle behaviors. This is all controlled by a proper use of the
Discrete Event Simulation (DES) kernel in Sparta using precedence (see
<<using_precedence>>).

==== Modeling Differences [[modeling_differences]]

gem5 o3 CPU model is not event driven. In fact, it is a PSNS model
that is explicitly controlled by 1 event called by the gem5 framework
(examine the tick method and observe the explicit advancement
calls). The topology in the o3 model is fixed with each component
being explicitly advanced in the order the modeler thinks is
appropriate. This hard coded topology and explicit advancement is very
restrictive especially if attempting a 0-cycle non-PSNS behavior
between blocks or even exploring multiple topologies.

Sparta allows for 0-cycle communication between blocks via its
precedence framework and allows the modeler to freely build unit
blocks that can be dynamically connected enabling the exploration of
different topologies at runtime. See an example of multiple topology
definitions in the {url-olympia}[Olympia] performance model `arches` directory.

When properly connected, the Sparta framework will automatically
determine the proper order in which a pipeline is to be advanced. The
framework is aware (or can be made aware) of 0-cycle dependency chains
and ensure the ordering is adhered. The framework can also ensure
cyclic behavior was not introduced which can lead to infinite loop
conditions.

==== Modeling Performance and Use

At the time of this book's writing, gem5's performance was considered
very slow. This lack of simulation performance is mitigated by using
SimPoint analysis to find relative hotspots in simulation and to use
checkpoints to only analyze those areas. This methodology certainly
mitigates that performance issue.

Modelers who build custom CPU models using Sparta roughly follow the
same pattern of workload analysis, but instead of running checkpoints,
modelers typically create {url-stf}[STF] traces using the same
SimPoint analysis flow and an instrumented functional model. This
allows a modeler using Sparta to not care about functional accuracy
but focus instead on performance analysis of his/her device.

Finally, if the event driven mechanism is properly utilized (i.e. only
schedule events when there is work to do), the modeler has complete
control of the performance of the simulation. The current design of
the o3 model does not allow that.

==== Summary

gem5 has advantages over Sparta; Sparta has advantages over
gem5. Either way the modeler must decide based on his/her use case and
intended purpose.

If the use case is fast CPU/SoC/GPU designs and tradeoffs with
multiple topology explorations and different architectures with
precise event handling, Sparta might be the better choice.

If the use case is rough performance adherence with full functional
support and not much detailed exploration via topological differences
or timing flows, then gem5 will be a good start.

=== Types of Simulation
include::attributes.adoc[]

Before diving into Sparta itself, understanding the landscape of
simulation styles that performance architects use provides useful
context.  Sparta is designed primarily for cycle-based simulation but
is flexible enough to support other approaches.

==== Cycle-Based Simulation

In cycle-based simulation, the simulator advances one clock cycle at a
time.  Each unit in the model -- fetch, decode, execute, and so on --
performs its work for the current cycle, passes data to downstream
units, and waits for the next cycle.  This is the dominant style for
CPU and GPU performance modeling because Sparta maps naturally to the
way digital hardware actually operates.

Sparta's scheduler and event system are designed around this model.
Units communicate through typed ports, events fire at specific cycle
offsets, and the framework guarantees a deterministic execution order
within each cycle.

==== Event-Driven Simulation

In event-driven simulation, the simulator does not advance by fixed
clock ticks.  Instead, Sparta maintains a queue of future events sorted by
time and jumps directly from one event to the next, skipping periods
of inactivity.  This can be significantly faster than cycle-based
simulation for sparse workloads where most units are idle most of the
time.

=== What Is in This Book
include::attributes.adoc[]

This book covers the Sparta framework from the ground up, starting
with simulation construction and working through communication,
scheduling, statistics, and analysis.

*Chapter 2 -- Simulation Construction* covers the class hierarchy that
makes up a Sparta simulation: `TreeNode`, `ResourceTreeNode`, `Unit`,
and `Resource`.  The chapter explains the four phases of simulation
construction -- Build, Configure, Instantiate, and Bind -- and shows
how to wire them together into a running simulator.

Throughout this book, examples draw from {url-olympia}[Olympia], the
open source RISC-V out-of-order performance model maintained under
RISC-V International.  Olympia represents a complete,
production-quality Sparta simulator and is the recommended companion
codebase for readers who want to see every concept applied at scale.
The full source for Sparta, Olympia, and the rest of the modeling
toolchain is available in the {url-map}[MAP repository] under the
Sparcians organization.
