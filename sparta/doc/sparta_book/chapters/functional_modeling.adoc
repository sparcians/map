= Sparta For Functional Modeling
include::attributes.adoc[]

Sparta is commonly associated with cycle-accurate performance modeling,
but the framework is equally well suited to *functional modeling* --
simulators that execute instructions correctly without modeling every
pipeline stage at cycle granularity.  Functional models are used for
ISA validation, software bring-up, fast workload profiling, and as the
reference model in co-simulation setups.

The key insight is that Sparta's infrastructure -- tree construction,
parameters, statistics, logging, reporting, and the scheduler -- is
completely independent of modeling granularity.  A functional model
uses the same `sparta::app::Simulation` flow, the same
`sparta::Unit`-based decomposition, and the same command line
integration as a cycle-accurate model.  What changes is how the
scheduler advances time and how much work each event represents.

{url-pegasus}[Pegasus] is the reference functional model for this
chapter.  Pegasus is an open source RISC-V functional simulator built
entirely on Sparta, and is available as a standalone repository at
`https://github.com/sparcians/pegasus`.

== Registers and RegisterSets
include::attributes.adoc[]

Architectural state -- integer registers, floating point registers,
CSRs, and vector registers -- is modeled using
{url-registerset}[`sparta::RegisterSet`] and
{url-register}[`sparta::Register`].  These classes are first-class
Sparta tree nodes, meaning registers appear in the simulation tree,
are automatically checkpointable, and can be read and written from
Python or from the command line during debugging.

=== sparta::RegisterBase::Definition

Each register is described by a
`sparta::RegisterBase::Definition` struct that specifies the register's
name, group, ID, size, and initial value.  Definitions are typically
loaded from a JSON file rather than hardcoded, allowing the register
file to be updated without recompiling.

[source,cpp]
----
// A minimal register definition in C++ (more commonly loaded from JSON)
static const sparta::RegisterBase::Definition INT_REG_DEFS[] = {
    //  name,  group_num,  group_name,  id,  size_bits,  initial_value,  desc
    { "x0",  1, "int",  0, 64, 0, "Zero register -- always reads 0" },
    { "x1",  1, "int",  1, 64, 0, "Return address" },
    { "x2",  1, "int",  2, 64, 0, "Stack pointer" },
    // ... remaining registers
    sparta::RegisterBase::DEFINITION_END
};
----

In Pegasus, register definitions are loaded from JSON files at
runtime.  This allows the same simulator binary to support different
ISA extensions by loading different register definition files:

[source,cpp]
----
// pegasus/arch/RegisterSet.hpp
class RegisterSet : public sparta::RegisterSet
{
public:
    // Create a RegisterSet from a JSON file
    static std::unique_ptr<RegisterSet> create(sparta::TreeNode* parent,
                                               const std::string & register_defns_json,
                                               const std::string & name = "regs")
    {
        auto defns = std::make_unique<RegisterDefnsFromJSON>(register_defns_json);
        return std::make_unique<RegisterSet>(parent, std::move(defns), name);
    }
};
----

=== Creating a RegisterSet

A `RegisterSet` is constructed as a child of a tree node, making
registers visible in the simulation tree under
`top.<unit>.regs.<register_name>`:

[source,cpp]
----
// In the unit constructor or onBindTreeEarly_()
reg_set_ = RegisterSet::create(getContainer(),
                                uarch_file_path_ + "/rv64gc/reg_defs.json",
                                "regs");
----

The `RegisterSet` owns the register definitions and must outlive all
`sparta::Register` pointers obtained from it.

=== Reading and Writing Registers

Individual registers are accessed by name or by register number.
The `sparta::Register` class provides typed read/write operations:

[source,cpp]
----
// Read a 64-bit integer register by number
uint64_t val = reg_set_->getRegister(reg_num)->read<uint64_t>();

// Write a 64-bit integer register by number
reg_set_->getRegister(reg_num)->write<uint64_t>(new_val);

// Access by name
auto* sp = reg_set_->getRegister("x2");
uint64_t stack_ptr = sp->read<uint64_t>();
----

Pegasus wraps these into convenience methods on its `RegisterSet`
derivative to reduce boilerplate at the call site:

[source,cpp]
----
// pegasus/arch/RegisterSet.hpp
template <typename T>
inline void writeRegister(uint32_t reg_num, T reg_value, uint32_t idx = 0)
{
    getRegister(reg_num)->write<T>(reg_value, idx);
}

template <typename T>
inline T readRegister(uint32_t reg_num, uint32_t idx = 0)
{
    return getRegister(reg_num)->read<T>(idx);
}
----

The `idx` parameter addresses sub-elements within a register.  For
scalar registers `idx` is always 0.  For vector registers, `idx`
selects the element within the register (e.g., a 512-bit vector
register with 8 `uint64_t` elements uses `idx` 0 through 7).

=== Register Fields

`sparta::Register` supports named bit fields within a register.
Fields are defined in the `RegisterBase::Definition` as a list of
`Field::Definition` structs:

[source,cpp]
----
static const sparta::RegisterBase::Field::Definition MSTATUS_FIELDS[] = {
    { "MIE",  3, 3,  false, "Machine Interrupt Enable" },
    { "MPIE", 7, 7,  false, "Machine Previous Interrupt Enable" },
    { "MPP",  11, 12, false, "Machine Previous Privilege" },
    sparta::RegisterBase::Field::DEFINITION_END
};
----

Fields can be read and written by name, and appear in the simulation
tree under `top.<unit>.regs.<register>.fields.<field_name>`:

[source,cpp]
----
// Read the MPP field of mstatus
auto* mstatus = csr_set_->getRegister("mstatus");
uint64_t mpp = mstatus->getField("MPP").read();

// Write the MIE field
mstatus->getField("MIE").write(1);
----

=== Multiple Register Groups

A single hart typically has several register groups: integer, floating
point, vector, and CSRs.  Each group is a separate `RegisterSet`
instance, attached as a child of the hart's tree node:

[source,cpp]
----
// In PegasusState constructor
int_regs_  = RegisterSet::create(hart_tn, arch_path + "/int_regs.json",  "int_regs");
fp_regs_   = RegisterSet::create(hart_tn, arch_path + "/fp_regs.json",   "fp_regs");
vec_regs_  = RegisterSet::create(hart_tn, arch_path + "/vec_regs.json",  "vec_regs");
csr_regs_  = RegisterSet::create(hart_tn, arch_path + "/csr_regs.json",  "csr_regs");
----

The group number in the `RegisterBase::Definition` identifies which
group a register belongs to when definitions from multiple groups are
combined.

== Sparta Memory Interface APIs
include::attributes.adoc[]

Sparta provides a layered memory interface that separates the logical
memory map from the physical storage backing it.  The key classes are:

[cols="1,2"]
|===
| Class | Purpose

| `sparta::memory::MemoryObject`
| Raw block of addressable memory storage.  Backed by a flat byte
  array.  Supports fill values for uninitialized regions.

| `sparta::memory::BlockingMemoryIF`
| Abstract interface for synchronous (blocking) memory access.
  Defines `tryRead()` and `tryWrite()`.  All memory accesses go
  through this interface.

| `sparta::memory::BlockingMemoryIFNode`
| A `BlockingMemoryIF` that is also a Sparta tree node.  Supports
  post-read and post-write callbacks for observers (co-simulation,
  trace generation, etc.)

| `sparta::memory::SimpleMemoryMapNode`
| A `BlockingMemoryIFNode` that maps address ranges to backing
  `MemoryObject` instances.  Acts as the system memory map.
|===

=== Setting Up a Memory Map

A typical functional model creates a `SimpleMemoryMapNode` as the
system memory map, then creates one or more `MemoryObject` instances
and registers them at their physical address ranges.  Pegasus does
this in `PegasusSystem`:

[source,cpp]
----
#include "sparta/memory/SimpleMemoryMapNode.hpp"
#include "sparta/memory/MemoryObject.hpp"

// Create the system memory map
memory_map_ = std::make_unique<sparta::memory::SimpleMemoryMapNode>(
    sys_node,
    "memory_map",
    sparta::TreeNode::GROUP_NAME_NONE,
    sparta::TreeNode::GROUP_IDX_NONE,
    "Pegasus System Memory Map",
    PEGASUS_SYSTEM_BLOCK_SIZE,   // block size: 4K
    PEGASUS_SYSTEM_TOTAL_MEMORY  // total address space: 8 EB
);

// Create a backing memory object for the DRAM region
auto dram = std::make_unique<sparta::memory::MemoryObject>(
    sys_node,
    BLOCK_SIZE,
    DRAM_SIZE,
    "DRAM",
    MEMORY_FILL_VALUE  // fill uninitialized bytes with this value
);

// Register the DRAM at its physical base address
memory_map_->addMapping(DRAM_BASE_ADDR, DRAM_BASE_ADDR + DRAM_SIZE,
                         dram.get(), 0 /*offset*/);
memory_objects_.emplace_back(std::move(dram));
----

=== Reading and Writing Memory

All memory accesses go through `BlockingMemoryIF::tryRead()` and
`tryWrite()`.  These return `bool` -- `true` on success, `false` if
the address is unmapped or the access is out of range:

[source,cpp]
----
// Read 8 bytes from physical address paddr
std::vector<uint8_t> buffer(8);
const bool success = memory->tryRead(paddr, 8, buffer.data());
sparta_assert(success, "Unmapped memory read at 0x" << std::hex << paddr);

// Write a uint64_t value to physical address paddr
const uint64_t value = 0xDEADBEEFCAFEBABEULL;
const std::vector<uint8_t> buf(reinterpret_cast<const uint8_t*>(&value),
                                reinterpret_cast<const uint8_t*>(&value) + 8);
const bool ok = memory->tryWrite(paddr, 8, buf.data());
----

Pegasus wraps these into typed templates that handle the
`std::vector<uint8_t>` buffer management automatically:

[source,cpp]
----
// Read a uint64_t from a translated physical address
template <typename MemoryType>
bool PegasusState::readMemory(const Addr paddr,
                               std::vector<uint8_t> & buffer,
                               const MemAccessSource source)
{
    const size_t size = sizeof(MemoryType);
    buffer.resize(size);
    return pegasus_core_->getMemory()->tryRead(paddr, size, buffer.data());
}
----

=== Memory Observers: Post-Read and Post-Write Callbacks

`BlockingMemoryIFNode` supports registering callbacks that fire after
every read or write.  These are used in Pegasus for co-simulation
verification and STF trace generation:

[source,cpp]
----
// Register post-write and post-read callbacks on the memory map node
void Observer::registerMemoryCallbacks(sparta::memory::BlockingMemoryIFNode* mem_node)
{
    mem_node->getPostWriteNotificationSource().REGISTER_FOR_THIS(postMemWrite_);
    mem_node->getPostReadNotificationSource().REGISTER_FOR_THIS(postMemRead_);
}

void Observer::postMemWrite_(
    const sparta::memory::BlockingMemoryIFNode::PostWriteAccess & data)
{
    // data.addr    -- physical address written
    // data.size    -- number of bytes
    // data.data    -- pointer to the bytes written
    // data.attempt -- pointer to the attempted write data (may differ on fault)
    recordMemoryWrite_(data.addr, data.size, data.data);
}
----

The callback mechanism integrates naturally with Sparta's notification
system.  Memory callbacks appear in the simulation tree and can be
enabled or disabled at runtime without modifying the model.

=== Memory-Mapped Devices

`BlockingMemoryIF` is also the base class for memory-mapped devices.
A device implements `tryRead_()` and `tryWrite_()` and registers
itself at its MMIO address range in the memory map.  Pegasus uses
this for its UART and magic memory device:

[source,cpp]
----
// SimpleUART derives from both sparta::Unit and sparta::memory::BlockingMemoryIF
class SimpleUART : public sparta::Unit,
                   public sparta::memory::BlockingMemoryIF
{
    // tryRead_ and tryWrite_ handle UART register accesses
    bool tryRead_(addr_t addr, addr_t size, uint8_t* buf,
                  const void* in, void* out) override;
    bool tryWrite_(addr_t addr, addr_t size, const uint8_t* buf,
                   const void* in, void* out) override;
};
----

The memory map routes accesses to the correct device based on address,
making the MMIO integration transparent to the instruction execution
path -- a load to a UART address calls `SimpleUART::tryRead_()` the
same way a load to DRAM calls `MemoryObject::tryRead_()`.

== Sparta Checkpointing
include::attributes.adoc[]

Sparta supports checkpointing of all functional state -- registers,
memory, and any modeler-defined state that derives from
{url-archdata}[`sparta::ArchData`].  Checkpoints are binary files and
are not human readable.  The primary use cases in functional modeling
are:

- Saving state after OS boot so subsequent runs can skip the boot phase
- Saving state at a region-of-interest entry point for repeated detailed analysis
- Co-simulation rollback when the performance model and functional model diverge

=== The Checkpointer Classes

Sparta provides two checkpointer implementations:

{url-fastcheckpointer}[`sparta::serialization::checkpoint::FastCheckpointer`]::
In-memory checkpointing using delta-checkpoint trees.  Only stores
state that has changed since the last checkpoint, making saves and
restores very fast.  Checkpoints are lost when the process exits.

{url-persistentcheckpointer}[`sparta::serialization::checkpoint::PersistentFastCheckpointer`]::
Extends `FastCheckpointer` to persist checkpoints to disk.  Allows
checkpoints to be saved during one run and restored in a subsequent
run.

Both implement the
{url-checkpointer}[`sparta::serialization::checkpoint::Checkpointer`]
interface, providing an ID-based API for saving and restoring
simulation state.

=== What Is Automatically Checkpointed

Any object that derives from `sparta::ArchData` is automatically
included in checkpoints.  The Sparta framework classes that derive
from `sparta::ArchData` include:

- `sparta::Register` -- all register values
- `sparta::memory::MemoryObject` -- all memory contents
- `sparta::Counter` and `sparta::StatisticDef` -- statistic values

This means a complete snapshot of register and memory state requires
no additional modeler code -- constructing the checkpointer and
calling `save()` is sufficient.

=== Adding Custom State to Checkpoints

Modeler-defined state that is not a `sparta::Register` or memory
object can be checkpointed by deriving from `sparta::ArchData` or by
using `sparta::ArchDataSegment`:

[source,cpp]
----
#include "sparta/functional/ArchData.hpp"

class PegasusState : public sparta::Unit
{
private:
    // Custom state that participates in checkpointing
    sparta::ArchData arch_data_{getContainer(), 64 /*line size*/};

    // A segment within the ArchData -- checkpointed automatically
    sparta::ArchDataSegment pc_segment_{&arch_data_, sizeof(uint64_t),
                                         sparta::ArchDataSegment::INVALID_ID};
    uint64_t& pc_ = *reinterpret_cast<uint64_t*>(pc_segment_.getLine()->getRawDataPtr(0));
};
----

// AUTHOR NOTE: Please verify the ArchDataSegment usage pattern above --
// the exact API for registering custom state with the checkpointer
// may differ from what is shown.  The Pegasus source uses RegisterSet
// (which derives from ArchData) for all checkpointable register state,
// so the raw ArchDataSegment pattern may not appear in Pegasus directly.

=== Using the Checkpointer

The `FastCheckpointer` is constructed after the tree is finalized,
takes the root tree node, and snapshots all `ArchData` descendants:

[source,cpp]
----
#include "sparta/serialization/checkpoint/FastCheckpointer.hpp"
#include "sparta/serialization/checkpoint/PersistentFastCheckpointer.hpp"

// After finalizeTree() -- create a checkpointer
sparta::serialization::checkpoint::PersistentFastCheckpointer checkpointer(
    *sim.getRoot(),   // root of the tree to checkpoint
    checkpoint_dir    // directory for checkpoint files
);

// Save a checkpoint -- returns a checkpoint ID
const auto chk_id = checkpointer.createCheckpoint();
ILOG("Checkpoint saved: id=" << chk_id);

// ... run more simulation ...

// Restore to a previous checkpoint
checkpointer.loadCheckpoint(chk_id);
ILOG("Checkpoint restored: id=" << chk_id);
----

In Pegasus, the `CherryPickFastCheckpointer` (a wrapper around
`FastCheckpointer`) is used in the co-simulation pipeline to support
branch rollback -- when the performance model and functional model
diverge, the functional model rolls back to the last committed
checkpoint and replays from there.

=== Triggering Checkpoints from Model Events

Checkpoints are typically triggered by model-level conditions rather
than at fixed intervals.  Common trigger patterns in functional models:

[source,cpp]
----
// Trigger: instruction count
if (sim_state_.inst_count == checkpoint_icount_)
{
    const auto id = checkpointer_->createCheckpoint();
    ILOG("Checkpoint at " << checkpoint_icount_ << " instructions: id=" << id);
}

// Trigger: PC value (e.g., Linux login prompt detection)
if (state->getPc() == login_prompt_pc_)
{
    checkpointer_->createCheckpoint();
    ILOG("Linux boot complete -- checkpoint saved");
}

// Trigger: specific instruction opcode
if (inst->getMnemonic() == "ecall" && a7_reg == SYS_checkpoint)
{
    checkpointer_->createCheckpoint();
}
----

== Using the Scheduling Mechanism
include::attributes.adoc[]

The Sparta scheduler is the engine that drives execution in a
functional model.  Unlike a cycle-accurate model where each pipeline
stage has its own events that fire every cycle, a functional model
uses a small number of events that each represent a large quantum of
work.

=== The Quantum Pattern

In Pegasus, a single event -- `ev_advance_sim_` -- drives the entire
simulation loop.  Each firing of this event executes one *quantum* of
instructions.  The quantum size is a parameter on the hart state:

[source,cpp]
----
PARAMETER(uint64_t, quantum, 500,
          "Instruction quantum size -- number of instructions "
          "executed per scheduler event")
----

Setting `quantum = 1` executes one instruction per scheduler event --
instruction-accurate simulation with the finest granularity.  Setting
`quantum = 1000000` executes one million instructions per event,
advancing simulated time by one million cycles per scheduler step.

The event is primed by a `StartupEvent` before cycle 0 and
reschedules itself at the end of each quantum:

[source,cpp]
----
// Primed once at startup
sparta::StartupEvent(core_tn, CREATE_SPARTA_HANDLER(PegasusCore, advanceSim_));

void PegasusCore::advanceSim_()
{
    PegasusState* state = threads_[current_hart_id_];

    // Execute instructions until quantum is exhausted or a pause
    // condition is encountered
    ActionGroup* next_action_group = fetch->getActionGroup();
    while (next_action_group)
    {
        next_action_group = next_action_group->execute(state);
    }

    // ... handle pause reasons ...

    // Reschedule at the cycle count after this quantum
    const uint64_t current_cycle = sim_state->cycles;
    if (threads_running_.any())
    {
        ev_advance_sim_.schedule(current_cycle - getClock()->currentCycle());
    }
}
----

The quantum boundary check fires inside the instruction execution path:

[source,cpp]
----
// After each instruction executes
if (sim_state_.inst_count % quantum_ == 0)
{
    pauseHart(SimPauseReason::QUANTUM);  // exits the ActionGroup loop
}
----

=== Yielding for Timed Events

Some instructions require the simulator to yield control to the Sparta
scheduler for a specific duration -- the RISC-V `PAUSE` hint
instruction is one example.  The hart is paused and a
`sparta::PayloadEvent` is scheduled at the appropriate future cycle:

[source,cpp]
----
// When a PAUSE instruction is detected
ev_pause_counter_expires_.preparePayload(current_hart_id_)
    ->schedule(current_cycle - getClock()->currentCycle()
               + pause_counter_duration_);

threads_running_.reset(current_hart_id_);  // hart is paused

// When the pause expires
void PegasusCore::pauseCounterExpires_(const HartId & hart_id)
{
    threads_[hart_id]->unpauseHart();
    threads_running_.set(hart_id);

    if (ev_advance_sim_.isScheduled() == false)
    {
        ev_advance_sim_.schedule();
    }
}
----

This pattern generalizes to any instruction that requires a timed
delay: yield by pausing the hart, schedule a `PayloadEvent` at the
future cycle, and resume in the callback.  The Sparta scheduler
advances time precisely to the event's scheduled cycle, making the
simulated timing exact even for large quantum sizes.

=== Multi-Hart Scheduling

Pegasus supports multiple hardware threads per core using a
round-robin schedule over a `std::bitset` of running harts.  After
each quantum, `advanceSim_()` advances to the next running hart:

[source,cpp]
----
// Round robin to next hart
++current_hart_id_;
current_hart_id_ = (current_hart_id_ == num_harts_) ? 0 : current_hart_id_;

if (threads_running_.any())
{
    ev_advance_sim_.schedule(current_cycle - getClock()->currentCycle());
}
----

When all harts stop -- `threads_running_` is empty -- `ev_advance_sim_`
is not rescheduled and the simulation ends naturally when the
scheduler runs out of events.

=== Multi-Mode Simulation

A common use case is running a long workload in fast functional mode,
then switching to a detailed cycle-accurate model for a specific
region of interest.  The functional model supports this through the
quantum parameter and pause mechanism.

The trigger conditions for switching modes are the same as checkpoint
triggers:

- **Instruction count** -- switch after N instructions
- **Instruction address** -- switch when the PC reaches a specific value
- **Instruction opcode** -- switch when a specific instruction executes

When the trigger fires, the functional model pauses the hart and
signals the cycle-accurate model to take over.  The mechanism for
transferring architectural state between models at the trigger point
is the modeler's responsibility -- a quantum manager that coordinates
this transfer is a natural extension of the pattern shown here.

NOTE: Multi-mode state transfer between a functional model and a
cycle-accurate model is not a built-in Sparta framework feature.  The
patterns shown above provide the infrastructure -- trigger detection
and hart pausing -- but the handoff protocol between models requires
modeler-defined coordination code.

== Summary
include::attributes.adoc[]

Sparta provides a complete infrastructure for functional modeling that
goes beyond the scheduler:

- {url-registerset}[`sparta::RegisterSet`] and `sparta::Register`
  provide structured, tree-visible, automatically checkpointable
  architectural register state loadable from JSON definitions
- The memory interface hierarchy (`MemoryObject` → `BlockingMemoryIF`
  → `SimpleMemoryMapNode`) provides a layered, observer-capable memory
  system that supports MMIO devices transparently
- `FastCheckpointer` and `PersistentFastCheckpointer` provide
  delta-based checkpointing of all `ArchData`-derived state with no
  modeler boilerplate for register and memory state
- The quantum scheduling pattern -- `StartupEvent` priming,
  `ev_advance_sim_` self-rescheduling, `PayloadEvent` for timed pauses
  -- gives precise simulated timing at any granularity from one
  instruction to millions

{url-pegasus}[Pegasus] demonstrates all of these patterns in a
complete, open source RISC-V functional simulator.  The Pegasus
source at `https://github.com/sparcians/pegasus` is the authoritative
reference for functional modeling with Sparta.
